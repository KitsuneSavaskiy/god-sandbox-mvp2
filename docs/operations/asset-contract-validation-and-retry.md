# Asset Contract Validation and Retry Plan

Sprint 10-B — Deterministic asset contract validator with retry plan output.

## Overview

The asset contract validator (`validate-asset-contract.mjs`) is a local quality gate
that checks generated PNG assets against machine-readable contracts.

It performs **structural checks only** (PNG header, dimensions, alpha channel).
Semantic quality (pose consistency, character identity, row order) requires human review.

## What the validator checks (hard gates)

All checks are deterministic: they read only the PNG file header (first 26 bytes) or verify file existence. No pixel decoding is performed.

| Check | Description |
|---|---|
| `file-exists` | Required asset file is present at the expected path |
| `png-signature` | First 8 bytes match the PNG magic number |
| `dimensions` | Canvas width and height match the contract exactly |
| `alpha-channel` | Color type includes alpha (type 4 = grayscale+alpha, type 6 = RGBA) |
| `canvas-size-consistency` | All expression files in a set share the same canvas size |

## What the validator does NOT check

- Semantic row quality (correct motion per row, consistent pose across frames)
- Character identity (does the sprite depict the intended character?)
- Pose sameness across expression variants
- Frame content or pixel-level margin compliance
- Whether the asset was generated by the approved pipeline

**identityConsistencyNeedsHumanReview** is set to `true` in the report for portrait
expression sets because semantic verification of character identity across expressions
cannot be automated.

## How to run

```bash
# Validate a combined PO preview sheet
npm run assetgen:validate-contract -- \
  --slug ryo \
  --contract resident-po-combined-preview-v1 \
  --asset-dir assets/generated/residents/ryo/incoming

# Validate canonical two-sheet sprites
npm run assetgen:validate-contract -- \
  --slug ryo \
  --contract resident-canonical-two-sheet-v1 \
  --asset-dir assets/generated/residents/ryo/incoming

# Validate portrait expressions
npm run assetgen:validate-contract -- \
  --slug ryo \
  --contract portrait-expression-set-v1 \
  --asset-dir assets/generated/residents/ryo/incoming

# Validate event standing expressions
npm run assetgen:validate-contract -- \
  --slug ryo \
  --contract event-standing-expression-set-v1 \
  --asset-dir assets/generated/residents/ryo/incoming

# Dry-run: print what would be validated, do not write files
npm run assetgen:validate-contract -- \
  --slug ryo \
  --contract resident-po-combined-preview-v1 \
  --asset-dir assets/generated/residents/ryo/incoming \
  --dry-run

# Help
npm run assetgen:validate-contract -- --help
```

## Contract IDs

| Contract ID | Expected files | Canvas size |
|---|---|---|
| `resident-canonical-two-sheet-v1` | `resident-sprite-sheet.png`, `resident-sprite-sheet-extended.png` | 1536×1872 each |
| `resident-po-combined-preview-v1` | `resident-sprite-sheet-combined.png` | 826×1904 |
| `portrait-expression-set-v1` | `expressions/neutral.png`, `happy.png`, `angry.png`, `sad.png`, `surprised.png` | Per-file (consistent) |
| `event-standing-expression-set-v1` | `event-expressions/neutral.png`, `happy.png`, `angry.png`, `sad.png`, `surprised.png`, `worried.png`, `determined.png`, `shocked.png` | Must be consistent across all files |

## Output files

Reports are written to `assets/generated/residents/<slug>/qa/` (gitignored).

### asset-contract-report.json

```json
{
  "candidateOnly": true,
  "slug": "ryo",
  "contractId": "resident-po-combined-preview-v1",
  "generatedAt": "2026-05-25T00:00:00.000Z",
  "passCount": 3,
  "failCount": 1,
  "checks": [
    { "check": "file-exists", "path": "...", "passed": true },
    { "check": "png-signature", "path": "...", "passed": true },
    { "check": "dimensions", "path": "...", "passed": false, "expected": "826x1904", "actual": "800x1900", "reason": "..." },
    { "check": "alpha-channel", "path": "...", "passed": true }
  ]
}
```

For portrait expression sets, an additional field is included:
```json
  "identityConsistencyNeedsHumanReview": true
```


### retry-plan.md

Human-readable version of `retry-plan.json`, formatted as markdown with
section headers per failure and plain-text action instructions.

## Retry plan scope values

Each failure in `retry-plan.json` has a `scope` field indicating what needs to be regenerated:

| Scope | Meaning |
|---|---|
| `full-sheet` | The entire sprite sheet must be regenerated |
| `row-only` | Only specific rows need to be regenerated |
| `frame-only` | Only specific frames need to be corrected |
| `expression-only` | Portrait expression file(s) need to be regenerated |
| `event-expression-only` | Event standing expression file(s) need to be regenerated |

## identityConsistencyNeedsHumanReview

This field is set to `true` in the report when the contract covers portrait
expression sets (`portrait-expression-set-v1`).

It signals that the automated validator has confirmed structural requirements
(files exist, valid PNG, alpha channel present) but **cannot verify**:

- Whether all expression files depict the same character
- Whether expressions match the reference portrait
- Whether expressions are visually plausible for the character's personality

A human reviewer must confirm character identity and expression quality before
promoting any expression set.

## Security

The validator refuses to write output to `public/art/**`. If `--asset-dir` or the
derived output path resolves under `public/art/`, the tool exits with a SECURITY error.

This prevents accidental promotion of candidate assets to production art paths.

## Exit codes

| Code | Meaning |
|---|---|
| 0 | Validator ran successfully (check `failCount` in the report for results) |
| 1 | Fatal error (bad arguments, unknown contract, security violation) |

Note: Exit code 0 does not mean all checks passed — it means the tool ran without error.
Read `failCount` in `asset-contract-report.json` to determine pass/fail status.
